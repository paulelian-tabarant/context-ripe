# US-0.1 — Walking Skeleton

**Slice**: 0 — Walking Skeleton  
**Status**: Ready for planning

## User Story

As a developer, I can deploy a minimal server + SPA to Railway and see a response from the server
rendered in the browser, so the full deployment pipeline is proven before any feature code lands.

## Context

This is the first deliverable. Its sole purpose is proving end-to-end connectivity — server boots,
SPA loads, they talk, it deploys. All code written here is intentionally minimal and will be replaced
or extended by subsequent slices.

Tech stack (ADR-004): TypeScript + Fastify (server), React + Vite (frontend), pnpm workspace.  
Deployment (ADR-013): Railway for hosting, GitHub Actions for CI/CD.  
Environments (ADR-013): staging + production as separate Railway services.

## Scope

**In scope:**
- Fastify server skeleton with a single `GET /api/health` endpoint
- React + Vite SPA that fetches `/api/health` and renders the response
- SPA served by Fastify's static file plugin
- Dockerfile for production build
- Railway configuration for staging and production services
- GitHub Actions: run lint, type checks, and tests on push; auto-deploy to staging on merge to
  `main`; deploy to production via manual trigger

**Out of scope:**
- Any business logic or real API endpoints (those come in later slices)
- Database setup (no SQLite yet)

## API Contract

```
GET /api/health
→ 200 { "status": "ok" }
```

## Deployment Strategy

- **On push (any branch)**: run lint, type checks, and tests (server + frontend)
- **On merge to `main`**: auto-deploy to staging; staging database resets on each deploy
- **Production**: manual trigger from the Railway dashboard after staging validation
- **Ad-hoc staging**: manual deploy from any branch to staging (for testing before merging)
- **Main branch**: protected — direct pushes are not allowed; changes land via pull requests only

## Acceptance Criteria

- [ ] `GET /api/health` returns `{ "status": "ok" }`
- [ ] SPA displays the message fetched from the server at the production URL
- [ ] SPA shows an error state if the server is unreachable
- [ ] CI runs lint, type checks, and tests on every push; failing CI blocks the deploy
- [ ] Merging to `main` automatically deploys to staging
- [ ] Production deployment requires a manual trigger
- [ ] Both staging and production URLs serve the SPA
